## Eventos: o que marca o início, o meio e o fim

Eventos são representados por **círculos** e indicam algo que "acontece"
durante o processo:

- **Evento de início** (círculo com borda fina): marca onde o processo
  começa — por exemplo, "pedido recebido"
- **Evento intermediário** (círculo de borda dupla fina): algo que
  acontece no meio do processo, como o recebimento de uma mensagem
- **Evento de fim** (círculo com borda grossa): marca onde o processo
  termina

## Tarefas: o que é feito de fato

Tarefas são representadas por **retângulos com cantos arredondados** e
descrevem uma atividade concreta, sempre começando com um verbo: "Analisar
solicitação", "Aprovar orçamento", "Enviar notificação".

Existem variações de tarefa (manual, de sistema, de usuário), mas para
quem está começando, o essencial é entender que cada retângulo representa
um passo específico do trabalho.

## Gateways: onde o processo toma decisões

Gateways são representados por **losangos** e controlam como o fluxo se
divide ou se junta:

- **Gateway exclusivo (X)**: representa uma decisão "ou isso, ou aquilo" —
  por exemplo, "pedido aprovado?" leva a caminhos diferentes conforme a
  resposta
- **Gateway paralelo (+)**: indica que múltiplas coisas acontecem ao mesmo
  tempo, sem depender uma da outra
- **Gateway inclusivo (O)**: permite que um ou mais caminhos sejam
  seguidos, dependendo das condições

## Fluxos de sequência: a ordem das coisas

As **setas** que conectam os elementos indicam a ordem em que o processo
acontece. É importante que o fluxo tenha sempre um caminho claro do início
ao fim — diagramas com setas soltas ou sem destino são um erro comum de
modelagem.

## Raias (pools e lanes): quem faz o quê

Quando um processo envolve mais de uma área ou pessoa, o BPMN usa
**raias** (lanes) dentro de uma **piscina** (pool) para mostrar quem é
responsável por cada etapa. Isso deixa visualmente claro onde o processo
"passa a bola" de uma área para outra — um dos pontos mais úteis do BPMN
para identificar gargalos entre departamentos.

## Um exemplo simples juntando tudo

Um processo de aprovação de despesa poderia ser lido assim: evento de
início ("despesa solicitada") → tarefa ("analisar valor") → gateway
exclusivo ("valor acima do limite?") → um caminho vai direto para
"aprovar" e o outro passa por uma tarefa extra ("enviar para gestor") →
os dois caminhos se encontram → evento de fim ("despesa processada").
